MIPROv2 interactive proposer goal
=================================

Goal
----

Split the proposer tool environment from the autonomous proposer loop so a
human, Codex, OpenEnv, a Codex app server, or any external agent can step through
a MIPROv2 proposer session interactively.

The explicit intention is collaborative testing. We should be able to start from
a live optimizer state or a before_proposer checkpoint, inspect the available
tools, call them one at a time, discuss the evidence, register hypotheses and
bets, preview or override queued rollouts, add candidate interventions, and then
commit or discard the session.

This creates a laboratory mode for proposer development. Instead of only editing
the autonomous proposer prompt and hoping it uses tools well, we can manually
drive the same environment and learn what a good proposer trace should look like.

Why this matters
----------------

Interactive sessions let us answer practical questions before baking behavior
into the autonomous proposer:

- Which tools are actually useful?
- Which tool outputs are too verbose, too lossy, or badly shaped?
- What does the proposer need in its instructions?
- When does TPE produce a good default queue?
- When should the proposer override the queue?
- Which hypotheses, bets, labels, and open-endedness scores help future
  candidate design?
- What does a gold proposer trajectory look like on Banking77, GSM8K, HOVER,
  HotpotQA, HealthBench Meta, or Ancient Greek?

Target split
------------

Optimizer harness:

- owns TPE
- owns train/eval loops
- owns candidate decoding
- owns ledger/checkpoints
- owns committed rollout queues
- decides when proposer sessions begin and when committed changes re-enter the
  optimizer

Proposer tool environment:

- owns a durable proposer session id
- exposes the tool catalog
- executes tool calls against a session snapshot
- records transcript/events
- supports checkpoint, commit, and discard
- can be driven by any proposer client

Proposer clients:

- current OpenAI/OpenEnv autonomous proposer
- Codex app server
- command-line human stepper
- external agent stepping through tool calls
- replay/eval proposer variants

Interactive flow
----------------

Example manual flow:

1. Start a proposer session from a before_proposer checkpoint.
2. List available tools.
3. Call get_grounding_summary.
4. Call query_candidates, query_rollouts, and list_sampled_train_rows.
5. Discuss what the evidence suggests.
6. Register or refine a hypothesis.
7. Register a falsifiable bet.
8. Preview the tentative TPE queue.
9. Override the queue if a different rollout slice is more informative.
10. Add a prompt-side or SFT-side candidate intervention.
11. Inspect the changed state.
12. Commit or discard the session.

CLI sketch
----------

The first external surface can be a simple CLI before HTTP/MCP:

```bash
mipro-proposer-session start --checkpoint path/to/checkpoint.json
mipro-proposer-session tools --session <session_id>
mipro-proposer-session call --session <session_id> get_grounding_summary '{}'
mipro-proposer-session call --session <session_id> query_rollouts '{"limit": 5}'
mipro-proposer-session call --session <session_id> register_hypothesis '{...}'
mipro-proposer-session call --session <session_id> preview_tpe_candidates '{"candidate_count": 8}'
mipro-proposer-session commit --session <session_id>
```

Core environment API sketch
---------------------------

```python
class MiproProposerEnvironment:
    def list_tools(self) -> dict: ...
    def snapshot(self) -> dict: ...
    async def call_tool(self, name: str, arguments: dict) -> dict: ...
    def checkpoint(self) -> dict: ...
    def commit(self) -> dict: ...
    def discard(self) -> dict: ...
```

Important semantics
-------------------

- The tool environment should be durable and resumable.
- Tool calls should be logged as events.
- Mutations should be session-local until commit.
- Commit should preserve both pre-session and post-session state.
- Discard should leave the optimizer untouched.
- Sessions should lock or version the base state so two proposers do not
  accidentally mutate the same optimizer state.
- Evidence-file access must remain scoped to the workspace root.

Implementation order
--------------------

1. Extract the current private action executor into a first-class
   MiproProposerEnvironment.
2. Make the existing OpenAI/OpenEnv proposer run against that environment with
   unchanged behavior.
3. Add session snapshots and event logs.
4. Add a CLI stepper.
5. Use interactive sessions with humans/Codex to discover better tool shapes and
   proposer instructions.
6. Add HTTP/MCP once the CLI and environment semantics feel right.
7. Add queue, hypothesis, bet, label, and open-endedness tools to the same
   environment.

This should come before over-hardening the autonomous proposer. Interactive use
will teach us which abstractions are useful and which ones are ceremony.

Detailed implementation plan
============================

Design principle
----------------

The proposer environment must become the stable product surface. The autonomous
OpenAI/OpenEnv loop should be just one client of that environment.

That means tool definitions, tool execution, session state, event logging,
checkpointing, and commit/discard semantics must live below any particular
proposer client. A human/Codex stepper and the autonomous proposer should see
the same tools and produce the same ledgered effects.

Target module layout
--------------------

Initial implementation should stay inside `synth_optimizers.miprov2.core`:

```text
core/
  proposer_environment.py
    MiproProposerEnvironment
    MiproProposerSession
    MiproProposerSessionSnapshot
    MiproProposerTool
    MiproProposerToolResult
    MiproProposerEvent
    MiproProposerCommitResult

  proposer_tools.py
    build_mipro_tool_catalog(...)
    execute_mipro_tool(...)
    individual tool handlers extracted from proposer_openenv.py

  proposer_sessions.py
    file-backed session store
    load/create/save/list sessions
    lock/version checks

  proposer_cli.py
    command-line stepper

  proposer_openenv.py
    autonomous OpenAI/Gemini/Groq/heuristic proposer clients
    provider-specific native tool calling
    prompt construction
```

`proposer_openenv.py` should eventually stop owning tool execution. It can keep
owning provider adapters and autonomous conversation policy.

V1 state ownership
------------------

The environment should start from a deep copy of optimizer/proposer state:

- compiled search space
- proposer context
- proposer config
- current instruction patches
- current demo patches
- optional queue/memory state

Tool calls mutate only the session-local copy. The live optimizer is not touched
until `commit`.

For checkpoint-backed interactive sessions, commit should initially write a
committed session artifact but not mutate a live optimizer process. That is
enough for replay, debugging, and gold-trace development. Live-process commit can
come after the file-backed version is reliable.

Session storage
---------------

Use file-backed sessions first. Put them under the optimizer artifact root:

```text
miprov2_artifacts/
  proposer_sessions/
    <session_id>/
      session.json
      events.jsonl
      pre_state.json
      current_state.json
      checkpoints/
        checkpoint_<NNN>.json
      committed_state.json
```

File-backed storage keeps the first implementation easy to inspect and robust
for interactive use. The SQLite ledger can index session ids and artifact paths,
but the complete session payload should remain readable as JSON.

Later, high-throughput runs can mirror session state into SQLite tables.

State versions and locks
------------------------

Every mutating tool call increments a `current_version` integer in the session.
Each event records the version before and after the call.

For file-backed sessions:

- acquire a session lock before a mutating call
- read `session.json`
- verify the expected version if provided
- apply the mutation
- write `current_state.json`
- append an event
- update `session.json`
- release the lock

Simple lock option:

```text
<session_dir>/.lock
```

The lock can store pid, hostname, created_at, and command. Avoid clever
distributed locking in v1.

Tool extraction plan
--------------------

Current private code in `proposer_openenv.py` should move in small pieces.

Step 1: create `MiproProposerToolState`.

Fields:

- compiled_space
- context
- config
- instruction_patches
- demo_patches
- optional memory_state
- optional queue_state

Step 2: move `_execute_action(...)` to `proposer_tools.py` as:

```python
def execute_mipro_tool(
    *,
    action: MiproOpenEnvAction,
    state: MiproProposerToolState,
) -> MiproProposerToolResult:
    ...
```

Step 3: keep `MiproOpenEnvAction` compatible for now. Do not rename the action
payload until the autonomous loop is stable against the new environment.

Step 4: make `run_openenv_react_proposer(...)` construct an in-memory
`MiproProposerEnvironment` and call `env.call_tool(...)` instead of private
`_execute_action(...)`.

Step 5: verify autonomous proposer traces are behaviorally equivalent.

The first refactor should preserve the existing tool names and results wherever
possible. Interactive mode should not start by changing proposer behavior.

Environment API
---------------

V1 Python API:

```python
@dataclass(slots=True)
class MiproProposerEnvironment:
    session: MiproProposerSession
    state: MiproProposerToolState
    store: MiproProposerSessionStore

    @classmethod
    def from_checkpoint(
        cls,
        checkpoint: Mapping[str, Any],
        *,
        session_root: str | Path,
        config: MiproOpenEnvProposerConfig | None = None,
        variant: MiproOpenEnvProposerVariant | None = None,
        actor_id: str = "interactive",
    ) -> "MiproProposerEnvironment": ...

    def list_tools(self) -> dict[str, Any]: ...
    def snapshot(self) -> dict[str, Any]: ...
    async def call_tool(
        self,
        name: str,
        arguments: Mapping[str, Any] | None = None,
        *,
        actor_id: str = "interactive",
        expected_version: int | None = None,
    ) -> dict[str, Any]: ...
    def checkpoint(self, *, actor_id: str = "interactive") -> dict[str, Any]: ...
    def commit(self, *, actor_id: str = "interactive") -> dict[str, Any]: ...
    def discard(self, *, actor_id: str = "interactive") -> dict[str, Any]: ...
```

V1 should also provide a pure in-memory constructor for the autonomous proposer:

```python
MiproProposerEnvironment.in_memory(
    compiled_space=...,
    context=...,
    config=...,
    variant=...,
)
```

That lets `run_openenv_react_proposer(...)` use the same environment without
forcing file-backed session storage during normal optimizer runs.

Interactive session transcript
------------------------------

Every tool call should append one JSONL event to `events.jsonl`. This event log
is the primary artifact for building gold proposer traces.

Example event:

```json
{
  "event_id": "evt_...",
  "session_id": "mps_...",
  "created_at": 1770000000.0,
  "event_type": "tool_call",
  "actor_id": "codex",
  "tool_name": "query_rollouts",
  "arguments": {"limit": 5},
  "state_version_before": 3,
  "state_version_after": 3,
  "result": {"status": "ok", "row_count": 5}
}
```

Mutating tools should include a compact `mutation_summary`:

```json
{
  "mutation_summary": {
    "instruction_patches_added": 1,
    "demo_patches_added": 0,
    "candidate_refs_added": ["candidate_..."]
  }
}
```

Commit semantics
----------------

Commit should be explicit and boring.

Checkpoint-backed session commit:

- write `committed_state.json`
- write `commit.json`
- mark `session.json.status = committed`
- do not mutate a live optimizer
- return new candidates/patches so replay harnesses can evaluate them

Live in-process session commit:

- verify base optimizer version still matches
- sync committed compiled search space into optimizer
- persist compiled space snapshot
- index session in the ledger
- mark session committed

Discard:

- mark `session.json.status = discarded`
- preserve event log and current state for debugging
- do not mutate optimizer state

This makes interactive work safe: experiments are never lost, but discarded
sessions do not affect the optimizer.

Autonomous proposer migration
-----------------------------

Current autonomous behavior:

```text
run_openenv_react_proposer
  build tool catalog
  build initial messages
  call agent.next_turn(...)
  execute action via _execute_action(...)
  update live messages/transcript
  return MiproOpenEnvProposerOutcome
```

Target autonomous behavior:

```text
run_openenv_react_proposer
  create in-memory MiproProposerEnvironment
  env.list_tools()
  build initial messages from env snapshot/tool catalog
  call agent.next_turn(...)
  env.call_tool(...)
  update live messages/transcript
  env.commit() or read env state
  return MiproOpenEnvProposerOutcome
```

The provider agents should not know whether tools are executed by direct
function calls, CLI, HTTP, or MCP. They only need tool definitions and tool
results.

External agent interaction
--------------------------

Codex/human interactive use does not need model-provider messages. It needs:

- a session id
- a list of tools
- JSON schemas
- JSON tool results
- a visible event log
- commit/discard

That means we can test proposer cognition without running `agent.next_turn(...)`
at all. Codex can inspect tool output, discuss it with the user, and decide the
next tool call.

This should become the preferred way to design new proposer tools. If a tool is
awkward for interactive Codex/human use, it is probably awkward for the
autonomous proposer too.

HTTP/MCP plan
-------------

Do HTTP/MCP after the CLI and Python environment are stable.

HTTP app:

- can be FastAPI
- should be optional dependency or eval/dev helper, not required for core import
- should reuse the file-backed session store
- should expose the same operations as CLI

MCP:

- expose `start_mipro_proposer_session`
- expose `list_mipro_proposer_tools`
- expose one generic `call_mipro_proposer_tool`
- expose `snapshot_mipro_proposer_session`
- expose `commit_mipro_proposer_session`
- expose `discard_mipro_proposer_session`

Avoid one MCP tool per proposer tool in v1. The proposer tool catalog is dynamic
and variant-dependent, so a generic call surface is easier to maintain.

Queue/hypothesis/bet integration
--------------------------------

Do not block the environment refactor on the full open-endedness stack. Add the
environment first, then add new tools.

Suggested order:

1. Existing evidence/search-space tools.
2. `preview_tpe_candidates`.
3. `register_hypothesis` and `append_hypothesis_adjustment`.
4. `register_bet` and `resolve_bet`.
5. `register_rollout_label_definition` and `query_rollouts_by_label`.
6. `score_rollout_open_endedness` and `query_open_ended_rollouts`.
7. `override_rollout_queue` and `commit_rollout_queue`.

Reason: `preview_tpe_candidates` is the smallest feature that validates the
harness/toolset split. Hypotheses/bets/labels become more useful once we can
interactively inspect and plan around the queue.

Compatibility requirements
--------------------------

- Existing full MIPROv2 runs should behave unchanged.
- Existing checkpoint replay should behave unchanged unless using the new
  interactive CLI.
- Existing tool names should remain stable for one release window.
- `MiproOpenEnvProposerVariant` should still be able to hide/reorder/reshape
  tools after the environment split.
- Checkpoint payloads should remain loadable. If new fields are added, they
  should be optional.
- The environment must support prompt candidates now and leave an explicit
  plugin path for SFT candidates.

Acceptance criteria for v1
--------------------------

Core:

- Start a file-backed session from an existing `before_proposer` checkpoint.
- List tools from the session.
- Call at least these tools through the environment:
  - get_grounding_summary
  - list_components
  - list_sampled_train_rows
  - query_candidates
  - query_rollouts
  - add_instruction_transform
  - finish
- See events appended to `events.jsonl`.
- Save and reload the session.
- Commit the session and get a committed-state artifact.
- Discard a different session and prove the source checkpoint is untouched.

Autonomous compatibility:

- Run the existing OpenAI/OpenEnv proposer through the environment.
- Confirm it still emits a `MiproOpenEnvProposerOutcome`.
- Confirm existing replay eval can call `run_proposer_from_checkpoint(...)`.

Interactive proof:

- Start a Banking77 checkpoint session.
- Codex/human calls a few read tools.
- Codex/human adds one instruction transform.
- Commit the session.
- Extract proposed candidates from the committed state.
- Evaluate them with the existing proposer checkpoint eval path.

Validation:

- `uv run --group dev ruff check src`
- `uv run --group dev ty check src`
- `python -m py_compile` for new CLI/environment modules
- one tiny Banking77 interactive smoke from an existing checkpoint

Risks and mitigations
---------------------

Risk: the refactor changes autonomous proposer behavior.

Mitigation: first move the executor behind an environment while preserving tool
names, results, and outcome shape. Avoid adding new tools in the same patch.

Risk: session state gets out of sync with optimizer state.

Mitigation: start with checkpoint-backed sessions, then add live commit with base
version checks.

Risk: external clients get unsafe file access.

Mitigation: preserve the existing workspace-root check for evidence files and
apply it inside the shared tool layer.

Risk: the environment becomes too generic too early.

Mitigation: keep MIPRO-specific names in v1. Generalize only after the
interactive flow works for Banking77.

Risk: CLI becomes a second implementation path.

Mitigation: CLI must call the Python environment/store only. No duplicate tool
execution logic in CLI.

Risk: hypotheses/bets/labels distract from the core refactor.

Mitigation: land the environment and CLI first, then add open-endedness tools as
separate commits.

First implementation slice
--------------------------

The first coding slice should be deliberately narrow:

1. Add `proposer_environment.py` with in-memory environment and state dataclasses.
2. Move `_execute_action(...)` into `proposer_tools.py` with minimal changes.
3. Make `run_openenv_react_proposer(...)` call the environment.
4. Add file-backed session store for checkpoint sessions.
5. Add CLI commands: `start`, `tools`, `snapshot`, `call`, `commit`, `discard`.
6. Smoke from an existing Banking77 checkpoint.

Defer until after the first slice:

- HTTP server
- MCP server
- queue overrides
- hypotheses
- bets
- labels
- open-endedness scoring
- SFT-specific tools

The goal of the first slice is to prove we can interactively step through the
existing proposer tool environment with no meaningful behavior drift.
